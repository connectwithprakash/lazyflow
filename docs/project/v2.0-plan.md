## Issue #283: Bidirectional Calendar Sync

### Architecture: Incremental Refactor

Keep existing CalendarSyncService primitives (debounce, loop prevention, completion policy, ID relink fallback). Extract internals into:
- `ForwardSyncEngine` / `ReverseSyncEngine`
- `SyncIdentityResolver`
- `SyncJournalStore`
- `SyncConflictResolver`

### Sync Journal Schema (Core Data, local-only)

```
CalendarSyncJournalEntity (syncable = NO)
├── id: UUID (PK)
├── operationKey: String (unique — direction + sourceID + action + timestamp)
├── directionRaw: Int16 (forward=0, reverse=1)
├── actionRaw: Int16 (create/update/delete/relink/skip/conflict/error)
├── statusRaw: Int16 (pending/applied/skipped/conflicted/failed)
├── taskID: UUID?
├── eventIdentifier: String?
├── externalIdentifier: String?
├── calendarIdentifier: String?
├── sourceModifiedAt: Date?
├── processedAt: Date?
├── payloadHash: String? (change fingerprint for dedup)
├── reasonCode: String?
├── retryCount: Int16
├── nextRetryAt: Date?
└── createdAt: Date

Indexes: (statusRaw, nextRetryAt), (taskID, processedAt DESC),
         (externalIdentifier, processedAt DESC), (payloadHash, sourceModifiedAt)
```

### PRs

| PR | Scope | SP | Key Files |
|----|-------|----|-----------|
| 283-1 | Refactor CalendarSyncService into forward/reverse pipelines (behavior-neutral) | 5 | CalendarSyncService, CalendarService, TaskService, new CalendarSyncPipeline, CalendarSyncDiff |
| 283-2 | Persistent sync journal (Core Data v4) + repository | 8 | New xcdatamodel version, PersistenceController, new CalendarSyncJournalStore |
| 283-3 | Inbound event→task creation + dedup + robust identity + deletion policy | 8 | CalendarSyncService, CalendarService, TaskService, Task model, new CalendarSyncIdentityResolver, CalendarSyncDedupService |

| 283-5 | Rollout flags + tests + observability | 3 | FeatureFlags, CalendarSyncServiceTests, new tests |

### Gaps Addressed
- ✅ Automatic inbound event→task creation (scoped to Lazyflow calendar first)
- ✅ Persistent sync journal replaces in-memory 10s cache
- ✅ Dedup checking via normalized fingerprint (time + title hash)
- ✅ Conflict resolution UI for ambiguous diffs
- ✅ Task deletion reverse-sync policy (configurable)
- ⏳ Intraday recurring: phase 1 = explicit unsupported + UX copy

## Issue #152: On-Device Knowledge Graph + GraphRAG

### Graph Schema (Core Data, local-only)

```
KnowledgeNodeEntity (syncable = NO)
├── id: UUID (PK)
├── normalizedKey: String (unique — e.g., "person:alice", "project:lazyflow")
├── displayName: String
├── nodeTypeRaw: Int16 (task/topic/person/project/time/place/custom)
├── salience: Double
├── confidence: Double
├── mentionCount: Int32
├── firstSeenAt: Date
├── lastSeenAt: Date
├── lastActivatedAt: Date?
└── metadataJSON: Binary?

KnowledgeEdgeEntity (syncable = NO)
├── id: UUID (PK)
├── edgeKey: String (unique — "sourceKey|relation|targetKey")
├── relationTypeRaw: Int16 (related_to/blocks/depends_on/belongs_to/scheduled_with/...)
├── sourceNode → KnowledgeNodeEntity (to-one)
├── targetNode → KnowledgeNodeEntity (to-one)
├── weight: Double
├── confidence: Double
├── evidenceCount: Int32
├── firstSeenAt: Date
└── lastSeenAt: Date

KnowledgeEvidenceEntity (syncable = NO)
└── Provenance to TaskEntity/QuickNoteEntity for explainability
```

### Migration Plan
1. New model version (lightweight migration compatible)
2. Keep existing entities untouched
3. Graph entities marked `syncable = NO` (no CloudKit payload growth)
4. Background backfill on first launch (bounded: 90 days of recent tasks)
5. Feature-flagged; fail open to current flat-context path

### PRs

| PR | Scope | SP | Key Files |
|----|-------|----|-----------|
| 152-1 | Graph persistence layer (Node/Edge + repository) + migration | 8 | New xcdatamodel version, PersistenceController, new KnowledgeGraphStore, KnowledgeGraphTypes, FeatureFlags |
| 152-2 | Extraction pipeline (NLTagger NER) + ingestion/upsert logic | 5 | New EntityExtractionService, KnowledgeGraphIngestionService, TaskService hooks, AIContextService |
| 152-3 | Graph retrieval + prompt compression + AI integrations | 8 | AIContextService, AIContext model, PrioritizationService, DailySummaryService, PromptTemplates, new GraphRetrievalService |

## Cross-Cutting Concerns

1. **Core Data versioning:** #283 (v4) and #152 (v5) both add entities — ship in order, not combined
2. **Settings IA dependency:** #285 should land before exposing new sync/graph controls
3. **Feature flags:** both #283 and #152 need kill-switches and staged enablement
4. **Observer contention:** sync, prioritization, and graph ingestion all react to data changes — centralize throttling
5. **CloudKit boundary:** sync journal and graph stay local-only
6. **QA load:** all three touch strings, snapshots, and UI tests — budget stabilization time
